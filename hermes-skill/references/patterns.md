# 思维问题与协作问题模式库

## 思维问题四模式

### ① 先查代码、后建假设

**现状：** 发现现象 → 直接读代码 → 在代码里找原因 → 陷入"读了很多代码但找不到问题"

**高效路径：** 发现现象 → 描述用户路径 → 提出假设 → 用假设反推验证方向

**关键识别：** 用户有猜测但不说出来，总是描述现象让 AI 自己去读代码

**典型信号：**
- 用户提示词中大量出现"帮我读这个文件"、"看看这段代码"
- 连续 3+ 次读同一文件但没有给出猜测
- 用户描述现象后直接问"这是为什么"，而非"我猜是不是因为 X"

### ② 对问题只定性不定量

**现状：** 说"偶尔"、"偶然"、"有时候"，不给概率和触发条件

**高效路径：** 给出"偶发约 1/N，连续操作 X 次以上时出现"

**关键识别：** AI 无法判断排查策略优先级，用户却不自知

**典型信号：**
- 提示词中出现频率词：偶尔、有时候、经常、总是、一直
- 没有给出操作次数、触发条件的量化描述
- 同一个问题反复出现，但每次描述都一样模糊

### ③ 把技术问题和业务问题混在一起问

**现状：** 问"debounce 能否实现"（技术方案），而非"如何防止重复点击重复请求"（业务问题）

**高效路径：** 先说业务问题 + 约束，让 AI 推荐方案，不让 AI 验证用户已想好的方案

**关键识别：** 技术选型先于问题定义

**典型信号：**
- 提示词中出现具体技术名词：debounce、throttle、useMemo、索引、缓存策略
- 用户已想好方案，但没先说业务背景和约束
- AI 实现了方案但业务效果仍不达预期（因为方案本身选错了）

### ④ 验证 vs 实现边界模糊

**现状：** 验证阶段问"怎么改"，或实现阶段还在问"是不是这里有问题"

**高效路径：** 明确说"帮我验证假设（不改代码）"或"假设确认了，现在帮我改"

**关键识别：** AI 交付物不是用户想要的，因为任务阶段不清晰

**典型信号：**
- 用户描述完问题后，AI 给出了修改方案，用户却说"等等，我还没让你改"
- 或相反：用户说"帮我改"，但 AI 先花很长时间解释问题原因
- 提示词中没有"验证"、"确认"、"改"等明确的动作词

## 协作问题五模式

### ① 大任务一次性塞入一个会话

**表现：** 把完整任务当一个会话，打断后全部重来

**关键数据：** 检查 `Request interrupted` 频率，占总会话数 ≥50% 需警示

**典型信号：**
- 同一项目在 7 天内有 5+ 个会话
- 多个会话都以 `Request interrupted` 结束
- 单会话 >1.5MB 且有 2+ 次中断

**改进建议：** 把大任务拆成 < 500KB 的子任务，每个会话专注一个目标

### ② 不记录中间状态，打断后无法续上

**表现：** 中断会话后，下一个会话从头读代码，没有交接信息

**关键识别：** 同一文件被重复读取 2+ 次

**典型信号：**
- 跨会话出现相同的"帮我读一下这个文件"
- 新会话的第一条消息就是长代码路径或文件描述
- 用户在新会话中重新描述的问题与上一个会话高度重合

**改进建议：** 每完成一个阶段，在项目目录写 `SESSION_SUMMARY.md`，记录已确认/未确认/下一步

### ③ 会话之间没有上下文传递

**表现：** 会话 A 确认了"排除 A，确认 B 有嫌疑"，会话 B 开始时又从头描述

**关键识别：** 同项目多会话中问题描述重复率 >70%

**典型信号：**
- 同项目多个会话的首条用户消息完全相同
- 不同会话中反复出现同一个问题的描述
- 用户说"上次我们讨论过"，但 AI 没有任何上下文

**改进建议：** 建立会话交接模板，关键结论用结构化方式记录

### ④ 工具没有分流

**表现：** 所有任务都往 Claude Code 塞，Cursor/Codex 空置

**关键数据：** Cursor 缓存目录有无可用会话，Codex 有无缓存记录

**典型信号：**
- Cursor 缓存目录（`~/Library/Application Support/Cursor/`）无本周修改文件
- Codex 缓存目录（`~/.claude/`）无会话记录
- Claude Code 会话中混杂了轻量任务（一次性代码生成）和重量任务（复杂排查）

**改进建议：**

| 场景 | 推荐工具 |
|------|---------|
| 排查偶发 bug、跨文件逻辑追踪 | Claude Code（长上下文） |
| 边写代码边问细节 | Cursor Chat |
| 一次性轻量代码生成 | Codex |
| 重复性自动化任务 | Cron + Claude Code |
| 需要并行两个独立任务 | 多个会话同时跑，不串行 |

### ⑤ 没有验收标准

**表现：** 改完代码后，AI 交付即结束，用户需自己判断对不对

**关键识别：** 用户提示词中无"怎样算改好了"的定义

**典型信号：**
- 用户提示词只有"帮我改"、"修复这个问题"，没有成功标准
- AI 说"改好了"，用户需要自己测试确认
- 改完后用户说"不对，不是这样"，说明验收标准不清晰

**改进建议：** 提示词中加入验收标准：
```
问题：点击按钮后 loading 不消失
期望：请求结束后 loading 消失，显示结果
已试：检查了网络请求，success 回调有执行
验收：连续点 5 次，每次 loading 都正常消失，无重复提交
```
